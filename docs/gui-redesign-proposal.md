# GUI 拆分与重设计建议

## 当前问题

- 现有 GUI 入口位于 `src/app/manual_probe_tool.py`，把工具台、坐标探针、识别工作台、自动战斗混在一个文件里，界面层和业务编排层耦合过重。
- `src/app/recognition_workbench.py` 虽然是 GUI 配套服务，但仍挂在 `app` 层命名空间下，不利于后续扩展成独立控制台。
- 当前 Tkinter 页面更接近“功能堆叠”，而不是“任务流”。对使用者来说，校准、识别、运行、复盘之间的切换成本偏高。

## 本次已做

- 新增独立 GUI 包：`src/gui`
- 将 GUI 主体迁移到：
  - `src/gui/manual_probe_tool.py`
  - `src/gui/recognition_workbench.py`
- 新增分层模块：
  - `src/gui/pages.py`
  - `src/gui/services.py`
  - `src/gui/models.py`
  - `src/gui/formatters.py`
  - `src/gui/theme.py`
- 保留旧路径兼容导出：
  - `src/app/manual_probe_tool.py`
  - `src/app/recognition_workbench.py`
- 首页已经改成总览台
- 自动战斗页已经改成运行控制台布局，而不是单一日志页
- 热更新监听已切到新的 `src/gui` 主结构

## 推荐的信息架构

建议把 GUI 从“多页工具箱”重排为“运行控制台”。

### 1. 总览台

- 当前实例
- 窗口绑定状态
- 最近一次识别结果
- 当前回合/状态机状态
- 最近一次动作反馈
- 异常和待处理告警

### 2. 校准台

- 左侧：按钮分组与状态
- 中部：按钮详情与坐标编辑
- 右侧：Before / After 截图对照
- 底部：人工判定记录

### 3. 识别工作台

- 左侧：识别模块列表与检测状态
- 中部：模块结果、区域参数、模板路径
- 右侧：Full / Region / Template 三联预览
- 顶部：单模块检测、全量检测、模板保存

### 4. 自动战斗台

- 顶部：启动、停止、运行模式
- 中部：当前战斗态、角色动作、宠物动作、反馈结果
- 底部：按回合折叠日志，而不是纯文本流水

## 更好的中期方案

如果目标是“自动化运行平台”，而不是“开发调试工具”，更推荐分两层：

- 短期：保留 Tkinter 作为本机工具台，继续承担校准和识别验证
- 中期：新增 Web 控制台，负责多实例运行、证据回放、策略切换、运行报表

原因：

- Tkinter 适合本地调试，不适合多实例监控和复杂复盘
- 运行平台天然更需要卡片化状态、时间线、截图证据流和实例矩阵
- Web UI 更容易做出高可读、高密度、可扩展的控制台体验

## 后续建议顺序

1. 继续把 `src/gui/manual_probe_tool.py` 拆成 `pages/`、`widgets/`、`services/`
2. 把“自动战斗页”从纯日志改成“状态卡片 + 回合时间线”
3. 为识别页增加模块置信度热力展示
4. 如果要做真正的平台，再单独起一个 Web 前端工程
