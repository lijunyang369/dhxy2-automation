## 1. 文档目的

## 3. 架构目标

### 3.1 一期目标

建立单窗口、固定分辨率下的自动化最小闭环：

1. 识别进入目标场景
2. 识别当前场景是否可操作
3. 按策略选择技能、操作或交互
4. 选择目标并提交动作
5. 等待界面反馈并迁移状态
6. 判定当前流程结束
7. 在异常条件下完成截图、日志和基础恢复

### 3.2 二期目标

- 支持角色策略配置化
- 支持更稳健的 OCR 识别
- 支持长时间运行
- 建立测试分层和回归能力
- 增强异常恢复能力

### 3.3 三期目标

- 支持多开窗口隔离
- 支持并发调度
- 支持更完善的证据回放
- 支持资源版本化与可维护迭代

## 4. 设计原则

- 先稳定再智能
- 先状态机再策略
- 先区域化识别再全屏识别
- 先配置驱动再硬编码分支
- 先模块边界再功能堆叠
- 先证据采集再追求极限速度
- 业务层禁止直接操作屏幕绝对坐标
- 策略层禁止直接调用底层自动化 API

## 4.1 模块化设计要求

模块化是本项目的强约束，不是风格偏好。

目标：

- 让单个能力可以被独立测试
- 让局部实现变更尽量局限在模块内部
- 让新增功能优先通过拼装模块完成，而不是继续堆脚本

设计要求：

- 模块边界必须围绕可独立验证的能力划分，例如配置加载、窗口绑定、按钮识别、观察结果构建、状态迁移、动作翻译、输入执行、恢复决策
- 每个模块必须有明确输入、输出和依赖，不允许通过隐式全局状态串联行为
- 模块依赖必须保持单向，禁止跨层直接调用下游内部实现
- 模块内部实现可以演进，但模块对外契约应尽量稳定
- 当需求变更时，优先新增或替换模块，而不是在已有入口中不断扩写条件分支

验收标准：

- 模块可以在脱离完整游戏链路的情况下被单独构造和测试
- 模块测试失败时，能快速定位到具体边界，而不是只能看到整链路失败
- 模块替换或局部重构后，不需要同步修改大量上层调用代码

## 5. 总体分层

项目建议采用七层架构。

### 5.1 平台接入层 `platform`

职责：

- 枚举和绑定游戏窗口
- 管理窗口聚焦、句柄状态和客户端区域
- 提供截图能力
- 提供鼠标键盘注入能力
- 提供坐标映射能力

关键约束：

- 坐标统一基于客户端坐标系
- 底层异常统一包装为窗口相关领域异常

### 5.2 感知层 `perception`

职责：

- 模板匹配
- 区域裁剪
- 像素与颜色特征识别
- OCR 调用
- 识别结果标准化输出

关键要求：

- 识别结果必须包含置信度
- 不允许只返回布尔值

### 5.3 领域层 `domain`

职责：

- 定义状态、动作、观察结果、异常类型
- 定义角色策略输入输出模型
- 定义自动化会话上下文

### 5.4 状态机层 `state_machine`

职责：

- 根据感知结果聚合当前状态
- 驱动状态迁移
- 提供恢复入口和失败出口

### 5.5 策略层 `policy`

职责：

- 根据角色配置和当前状态生成动作决策
- 管理技能优先级、目标选择规则和条件回退逻辑

### 5.6 执行层 `executor`

职责：

- 将领域动作翻译为具体操作序列
- 负责技能栏点击、目标选择、确认等流程执行
- 负责动作执行结果回传

### 5.7 编排运行层 `runtime` / `app`

职责：

- 管理主循环
- 管理超时、重试、恢复和终止
- 管理日志、截图、证据包和会话目录
- 管理测试运行与后续调度接入点

## 6. 核心调用链

建议保持单向依赖关系：

`platform -> perception -> domain -> state_machine -> policy -> executor -> runtime/app`

实施约束：

- 高层模块不得反向依赖底层实现细节
- 策略层只面向领域对象，不感知 Airtest 细节
- 运行层负责调度，不写业务判断
- 入口脚本只负责组合模块，不负责承载模块内部逻辑

## 7. 核心对象建议

建议在一开始定义少量稳定的核心对象，避免后续模块之间通过散乱字典传值。

建议对象：

- `WindowSession`
- `Frame`
- `MatchResult`
- `OCRResult`
- `AutomationObservation`
- `AutomationContext`
- `AutomationAction`
- `ActionPlan`
- `TransitionResult`
- `RecoveryDecision`

这些对象的目标是保证模块边界清晰、日志可追踪、测试可断言。

## 8. OCR 方案建议

当前建议：

- 架构上采用可插拔 OCR 接口
- 优先通过本机独立 OCR 服务 服务接入 PaddleOCR
- Tesseract 作为备选或补充方案

选择理由：

- 中文文本识别更贴近目标场景
- 游戏内按钮和提示文本更适合区域化中文 OCR
- 能隔离 PaddleOCR 与业务项目环境冲突
- 后续可根据性能和部署成本再做替换或混用

建议抽象接口：

- `read_text(region, profile)`
- `read_lines(region, profile)`
- `detect_text_boxes(region, profile)`

## 9. 测试与质量策略

建议在一期就纳入 pytest，而不是等脚本堆积后再补。

测试分层：

- 单元测试：状态机、策略选择、配置加载
- 识别测试：模板匹配和 OCR 区域样本验证
- 冒烟测试：最小闭环
- 回归测试：多种策略和异常分支
- 长稳测试：长时间运行与稳定性验证

Allure 建议作为二期增强，不作为当前阻塞项。

## 11. 首版落地约束

当前建议明确以下约束，作为后续实现的硬规范：

1. 业务层禁止直接写绝对屏幕坐标
2. 策略层禁止直接调用 Airtest 或 win32 输入 API
3. 所有识别结果必须带置信度和上下文
4. 所有关键动作必须留下日志和截图证据
5. 所有异常必须有分类和恢复策略
6. 所有区域、模板、阈值应逐步配置化
