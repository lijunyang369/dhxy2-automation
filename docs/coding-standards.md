# 《大话西游2》项目代码补充规范

## 1. 文档定位

本文档不是项目的独立全局代码规范。

本项目默认继承工作区通用规范：

- `D:\Codex\standards\global-coding-standards.md`

本文档只补充《大话西游2》战斗自动化项目特有的实现约束。

## 2. 项目补充约束

### 2.1 分层约束

- `policy` 不直接调用 Airtest 和 win32 API
- `executor` 不决定业务策略
- `state_machine` 不关心具体点击细节
- `perception` 不直接修改运行状态

### 2.2 识别约束

- 所有识别结果必须带置信度
- 不允许只返回布尔值作为最终识别结果
- OCR、模板匹配、区域识别都必须可留证

### 2.3 自动化约束

- 业务层禁止直接写绝对屏幕坐标
- 所有动作必须经由统一执行层发出
- 关键动作前后必须保留日志或截图证据

### 2.4 稳定性约束

- 异常不得静默吞掉
- 恢复流程必须可追踪
- 关键状态迁移必须记录日志
- 高风险动作必须可回放上下文

### 2.5 模块化设计约束

- 新功能优先以独立模块落地，不允许在单个脚本或单个大类中持续堆叠逻辑
- 每个模块必须职责单一，输入、输出和副作用边界清晰
- 模块之间只通过稳定接口或明确的数据对象交互，不允许跨模块直接读写内部状态
- 配置加载、窗口绑定、识别、状态判定、动作翻译、执行、恢复等能力必须可独立测试
- 模块设计必须支持替换内部实现而不破坏外部契约，优先做到“模块内改动，模块外无感”
- 高风险变更应先收口到模块边界内，避免一次改动横跨多个层级
- `scripts` 目录只负责组装模块和提供入口，不承担核心业务模块实现

### 2.6 可测试性约束

- 每个核心模块至少应有对应的单元测试或 smoke 测试入口
- 测试应优先覆盖模块边界和模块契约，而不只覆盖最终整链路结果
- 模块对外依赖应支持注入、替身或静态样本，避免测试必须依赖真实游戏窗口才能启动
- 当某个模块难以单独测试时，应优先重构模块边界，而不是继续增加集成脚本

## 3. 结论

本项目代码实现先遵守全局代码规范，再遵守本补充规范。

若两者冲突，以更严格者为准。
